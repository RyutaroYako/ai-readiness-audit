# コードベース AI-Readiness 評価スキル

AI（や新しいチームメンバー）がこのコードベースで作業するとき、「迷わず正しい判断を下せるか？」を評価する。
静的解析のメトリクス（行数、複雑度など）ではなく、**意味的な明確さ**を問う。

## 制約

- コードベースの調査には `Glob`、`Grep`、`Read` ツールを使うこと。`Bash` で `ls`・`grep`・`cat` 等を実行しない（ユーザーに不要な権限プロンプトを出さないため）

## 評価の進め方

1. まずコードベース全体の構造を把握する（ディレクトリ構成、主要ファイル、README、CLAUDE.md、ADR等のドキュメント）
2. 以下の7つの観点それぞれについて、具体的なコードやファイルを根拠に評価する
3. 各観点を以下の5段階で判定する
   - **5** 🟢 — AIが迷う余地がない。ドキュメント・コード共に明確
   - **4** 🟢 — 概ね明確。軽微な改善の余地がある程度
   - **3** 🟡 — 部分的に曖昧。AIが誤った判断をするリスクがある
   - **2** 🔴 — 曖昧な箇所が多い。AIが高確率で迷う
   - **1** 🔴 — 判断材料がほぼない。AIが正しく動ける見込みが低い
4. 最後に7観点の合計を100点満点に換算した総合スコアと優先度の高い改善提案をまとめる（計算式: 合計 ÷ 35 × 100、小数点以下四捨五入）

## 評価観点

### 1. 「どこに書くべきか」が一意に決まるか

新しい機能やモジュールを追加するとき、置き場所に迷わないか。

**確認ポイント:**
- ディレクトリ構造に一貫した整理原則があるか（機能別、レイヤー別、ドメイン別など）。混在していないか
- 既存のファイルの配置を見れば、新しいファイルをどこに作るべきかが推測できるか
- 似た役割のコードが複数の場所に散らばっていないか（例: バリデーションロジックが controller にも service にも model にもある）
- 「ここには何を置き、何を置かないか」が README やアーキテクチャドキュメントで説明されているか

**AIが迷う典型例:**
モジュール A では `src/features/xxx/` に機能をまとめているが、モジュール B では `src/pages/xxx/` と `src/hooks/xxx/` と `src/utils/xxx/` に分散している。AI は両方のパターンを見て、どちらに従うべきか判断できない。

### 2. 「どのパターンで書くべきか」が一意に決まるか

同じ種類の処理を実装するとき、お手本となるパターンが1つに定まっているか。

**確認ポイント:**
- エラーハンドリングの方法は統一されているか（例外を投げる / Result 型を返す / エラーコードを返す、が混在していないか）
- データの取得・永続化のパターンは統一されているか（直接 DB アクセス / Repository パターン / ORM 直呼び、が混在していないか）
- 状態管理の方法は統一されているか（フロントエンドなら: useState / Redux / Zustand / Context が場所によって異なっていないか）
- API のレスポンス形式、ログの出し方、認証・認可の扱い方は一貫しているか
- 「正解のパターン」が存在する場合、それがドキュメントまたはコード上のコメントで示されているか。古いパターンが残っている場合、それが「古い」とわかるようになっているか

**AIが迷う典型例:**
エラーハンドリングが try-catch のモジュールと `Result<T, E>` を返すモジュールが混在していて、AI は新しいモジュールでどちらを採用すべきかわからない。あるいは両方を混ぜた不整合なコードを生成する。

### 3. 命名がドメインの意図を伝えているか

コードの命名から、そのコードが「何のために存在するか」が読み取れるか。

**確認ポイント:**
- ビジネスドメインの用語がコード上で一貫して使われているか（例: 同じ概念を user / account / member と呼び分けていないか）
- `utils`、`helper`、`common`、`misc` のような意味の薄いモジュール名が多用されていないか
- 関数名から、その関数が何をするか（副作用を含むか、何を返すか）が推測できるか
- 型名やインターフェース名が、ドメインの制約を表現しているか（`string` ではなく `EmailAddress`、`number` ではなく `UserId` など）

**AIが迷う典型例:**
`processData()` という関数があるが、何のデータをどう処理するのかが名前から読み取れない。AI は呼び出し元を全部読まないと用途がわからず、類似の新しい処理を書くときに間違った場所に間違った実装を追加する。

### 4. 暗黙の前提が明文化されているか

「チームでは当然知っていること」がコードやドキュメントに残っているか。

**確認ポイント:**
- アーキテクチャ上の重要な判断（なぜこの技術を選んだか、なぜこの構造にしたか）が記録されているか（ADR など）
- 「やってはいけないこと」が明示されているか（例: この API は直接呼ばずに必ず Facade を経由する、このテーブルは直接 INSERT しない、など）
- 環境固有の制約（外部 API のレート制限、DB コネクションプールの上限、特定の処理順序の要件など）が、コード上のコメントまたはドキュメントに残っているか
- 歴史的経緯で「本来はこうすべきだが今はこうなっている」というコードに、その理由が書かれているか

**AIが迷う典型例:**
注文処理では在庫確認 → 決済 → 在庫引当の順序でなければならないが、その制約がどこにも書かれていない。AI は効率を考えて並列化したり順序を変えたりして、本番で不整合を起こすコードを生成する。

### 5. テストが「仕様」として機能しているか

テストコードを読めば、そのモジュールの期待される振る舞いがわかるか。

**確認ポイント:**
- テストの名前が振る舞いを記述しているか（`test_1` や `should work correctly` ではなく、`未ログインユーザーが注文するとエラーが返る` のような記述）
- テストが「実装の内部構造」ではなく「外から見た振る舞い」を検証しているか
- 正常系だけでなく、境界値やエラーケースがテストされているか
- テストを読めば、そのモジュールの「入力と出力の関係」が理解できるか
- テストが壊れたとき、何が壊れたのかがテスト名やエラーメッセージからわかるか

**AIが迷う典型例:**
テストが mock だらけで実装の内部構造に強く依存している。AI はテストを仕様として読めず、リファクタリングの際にテストが通らなくなる変更を避けようとして、本来すべき構造改善を行えない。

### 6. モジュール間の依存関係が明確か

モジュール間の関係が整理されていて、あるモジュールを変更したときの影響範囲が予測可能か。

**確認ポイント:**
- モジュール間の依存方向が一方向か（循環依存がないか）
- あるモジュールの public なインターフェース（型、関数のシグネチャ）が明確に定義されているか
- モジュールの内部実装の変更が、他のモジュールに波及しにくい構造になっているか
- 依存関係がコード上で明示的か（暗黙のグローバル状態や、実行時にしかわからない依存がないか）

**AIが迷う典型例:**
グローバルな状態（シングルトン、グローバル変数、暗黙的な DI コンテナ）に多くのモジュールが依存していて、AI はある変更の影響範囲を正確に把握できない。結果として安全に見える変更が、遠く離れた場所で副作用を引き起こす。

### 7. 例外・特殊ケースが管理されているか

「ここだけ違う」というコードが、なぜ違うのかが明確か。

**確認ポイント:**
- 共通パターンから外れているコードに、その理由がコメントやドキュメントで説明されているか
- 技術的負債として認識されている箇所が `TODO` / `FIXME` / `HACK` などでマークされ、その意図が記述されているか
- deprecated なコードパスや旧バージョンとの互換性のためのコードが、そうとわかるように管理されているか
- 特定の顧客・環境向けの分岐（feature flag、環境分岐）が整理されているか、散在していないか

**AIが迷う典型例:**
あるモジュールだけ異なるライブラリを使って同じことをしている。AI はそれを「このプロジェクトでは両方使ってよい」と解釈し、新しいコードでも混在させる。実際には移行途中で、新しいコードでは新しいライブラリのみを使うべきだった。
